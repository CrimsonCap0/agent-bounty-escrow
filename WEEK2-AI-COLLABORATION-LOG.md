## 一、这次协作做了什么

我和 AI 一起完成了 `Agent 任务托管` 的第一版。这个项目让发布者把 MON 赏金锁进智能合约，执行者接单并提交结果，发布者确认后由合约放款。

## 三、我删改和核查了什么

我没有把 AI 的第一版建议全部保留。

### 1. 收缩项目范围

讨论中出现过 DAO 仲裁、信誉积分、NFT、ERC20 支付、跨链和主网部署。我把这些都移出本周范围，只保留 MON 测试币托管和最小任务流程。原因很简单：当前最需要证明的是赏金能否按正确状态锁定、退款和放款，功能越多，越难查清问题。

### 2. 核对合约权限和资金路径

我重点检查了这些条件：

- 发布者不能接自己的任务。
- 只有接单者能提交结果。
- 只有发布者能确认放款。
- 开放状态的任务可以由发布者取消并退款。
- 放款前先把赏金清零并更新状态，降低重复调用风险。

这些检查不等于正式安全审计。目前合约没有争议仲裁，已接单但逾期的任务也缺少完整处理方案，所以不能当作生产级托管合约。

### 3. 人工完成链上操作

钱包连接、网络确认、部署签名和交易确认由我操作。AI 可以告诉我按钮和字段的含义，但它不应该替我决定是否签名。我还通过区块浏览器核对合约地址和部署交易，避免只根据前端的成功提示判断结果。

### 4. 检查提交内容和发布权限

我确认 README 和公开仓库里没有助记词、私钥或 API Key。学习日志推送后，我还追问 AI 是如何访问仓库的，确认它使用的是本机已经配置好的 Git 凭据，而不是掌握了我的 GitHub 密码或另行调用 GitHub API。

### 5. 没把“页面能打开”当成“项目已完成”

公开 Demo 可以加载，但 2026-07-14 检查时出现：

```text
Cannot read properties of null (reading 'taskCount')
```

因此我把项目状态保留为 MVP，而不是宣称完整流程已经可用。下一步要先修复读取错误，再做双钱包测试。

## 四、哪些不能交给 AI

### 钱包秘密和签名决定

助记词、私钥、钱包密码、API Key 不能发给 AI。每一次链上签名都要由我确认网络、合约地址、调用函数和金额。即使是测试网，也要保留这个习惯。

### 最终产品和范围判断

AI 可以列出很多功能，但不能替我决定哪些功能值得做。这次删掉复杂仲裁和 NFT，就是因为它们会分散本周目标。是否需要上链、服务谁、怎样算完成，最终由我负责。

### 安全结论

AI 可以指出常见风险、生成测试和辅助代码审查，但不能给合约出具“安全”结论。涉及真实资产前，仍需要完整测试、独立代码审查和专业审计。

### 事实与结果核验

AI 生成的合约地址、交易 hash、文档链接和运行说明都可能写错。我需要在区块浏览器、GitHub 和实际页面中逐项核对。线上出现 `taskCount` 错误也说明，代码生成和部署成功不等于真实用户流程已经跑通。

### 对外发布和权限

提交代码、推送仓库、开放文档权限都属于对外操作。即使使用的是本机已有凭据，也应该让我知道修改了哪些文件、推送到哪个分支，并保留可追踪的 commit。

## 五、我的结论

这次协作里，AI 更像一个能快速写草稿、查资料和陪我排错的开发搭档。它缩短了从想法到 Demo 的时间，但也容易把“已经生成代码”说成“已经解决问题”。

对我来说，合理分工是：AI 负责提出方案、生成初稿、解释和检查；我负责范围、签名、事实核验、安全边界和最终提交。下一步我会继续让 AI 协助定位 `taskCount` 错误和补测试，但只有在双钱包流程真实跑通并能从区块浏览器复查后，才把本周任务标为完成。
